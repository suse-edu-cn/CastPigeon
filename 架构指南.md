# 🤖 CastPigeon 架构与开发规范 (AI System Prompt)

## 0. 架构元数据 (Architecture Metadata)

* **Project Name**: CastPigeon
* **Description**: 一个极低延迟、无网传输、基于物理光学渲染（液态玻璃）的跨端文件传输信道。
* **Tech Stack**:
* UI 层: **Flutter / Dart** (利用 Impeller 引擎实现高级着色器与多端 UI 一致性)。
* 逻辑层: **KMP (Kotlin Multiplatform)** (纯 Kotlin，抛弃 JVM)。
* 通信层: **Dart FFI & Kotlin C-Exports (C ABI)** (零胶水代码，纯指针通信)。


* **Core Philosophy**: **UI 与逻辑绝对物理隔离。利用 Mock 机制保留 Flutter 热重载，利用 C ABI 榨干 Native 性能。**

## 1. 模块化与目录结构 (Directory Structure)

AI 生成代码时，必须严格遵守以下边界：

```text
/
├── kmp_core/               # 纯 Kotlin 逻辑层 (绝对禁止包含任何 UI 代码)
│   ├── build.gradle.kts    # 配置 macosX64, mingwX64 等 Native Target
│   ├── src/commonMain/     # 核心业务 (StateMachine, UdpDiscovery, Ktor)
│   └── src/nativeMain/     # FFI 暴露层 (带 @CName 的 C-Exports 函数)
│
└── flutter_app/            # Flutter 展现层 (绝对禁止包含核心业务逻辑)
    ├── lib/core/bridge/    # 跨语言通信抽象层 (Interface + Mock + FFI)
    ├── lib/ui/             # 液态玻璃特效与常规组件
    └── lib/main.dart       # 依赖注入入口

```

## 2. KMP 核心逻辑层规范 (KMP Constraints)

当 AI 编写 `kmp_core/` 目录下的代码时，必须遵守以下规则：

* **[FORBIDDEN] 禁用主线程调度器**: 绝对禁止使用 `Dispatchers.Main`。所有协程必须运行在 `Dispatchers.Default` 或 `Dispatchers.IO`。
* **[FORBIDDEN] 禁用 JVM 强绑定**: 绝对禁止引入仅支持 JVM 的依赖库。所有依赖必须支持 `linuxX64`, `mingwX64`, `macosX64`, `macosArm64`。
* **[STATE MANAGEMENT]**: 状态机（如 `ConnectionStateMachine`）必须使用 `StateFlow` 或 `SharedFlow` 暴露状态。

## 3. FFI 通信协议 (FFI Bridge Protocol)

当 AI 编写 `kmp_core/src/nativeMain/` (Kotlin) 和 `flutter_app/lib/core/bridge/` (Dart) 时，必须遵守以下三级通信协议：

### Protocol A: 同步调用 (Dart -> Kotlin)

* **规则**: 仅支持基本数据类型 (`Int`, `Double`, `Bool`)。
* **Kotlin 侧**: 必须使用 `@CName("func_name")` 注解。
* **Dart 侧**: 必须使用 `lookupFunction` 绑定。

### Protocol B: 异步状态推送 (Kotlin -> Dart)

* **规则**: 用于状态机状态变更。**禁止序列化**。枚举必须在 Kotlin 降维成 `Int`。
* **实现机制**:
1. Dart 初始化时创建 `ReceivePort`，将 `nativePort` (整型句柄) 传给 Kotlin。
2. Kotlin 层引入 `dart_api_dl.h`，在 Flow 收集到新状态时，使用 `Dart_PostCObject` 推送 `Int` 状态码。



### Protocol C: 复杂数据与对象列表传递

* **规则**: 用于传递 `UdpDevice` 列表等低频复杂数据。
* **实现机制**:
1. Kotlin 端使用 `kotlinx.serialization` 转为 JSON 字符串。
2. Kotlin 使用 `string.cstr.getPointer(nativeHeap)` 在 Native 堆分配内存。
3. **[CRITICAL] 内存释放**: Dart 端通过 `toDartString()` 解析完毕后，**必须立即**调用 Kotlin 导出的 `free_string_pointer(Pointer)` 方法释放内存。绝对不允许内存泄漏。



## 4. Flutter UI 层开发规范 (Flutter Constraints)

当 AI 编写 `flutter_app/` 目录下的代码时，必须遵循“三层依赖注入隔离”：

* **第一层：抽象契约 (Interface)**
所有 UI 只能依赖抽象类 `CastPigeonApi`。
```dart
abstract class CastPigeonApi {
  Stream<int> get connectionStateStream;
  Stream<String> get deviceListJsonStream;
  Future<bool> startDiscovery();
}

```


* **第二层：Mock 引擎 (Mock Engine)**
* **类名**: `MockCastApi implements CastPigeonApi`
* **作用**: 在纯 UI 开发期使用，通过 `Timer` 和假数据模拟底层状态。
* **目标**: 确保 UI 开发阶段享受 100% Hot Reload。


* **第三层：FFI 引擎 (FFI Engine)**
* **类名**: `FFICastApi implements CastPigeonApi`
* **作用**: 打包与真机联调时使用。
* **限制**: 此文件内封装所有的 `dart:ffi` 指针操作，UI 层 Widget 绝对禁止直接 `import 'dart:ffi'`。


* **依赖注入阀门 (DI Toggle)**
在 `main.dart` 中，通过顶级常量控制模式：
```dart
const bool isUiDevelopmentMode = true; // AI 编写 UI 时必须设为 true
final CastPigeonApi api = isUiDevelopmentMode ? MockCastApi() : FFICastApi();

```



## 5. 文件传输与 I/O 规范 (File Transfer Rule)

* **[CRITICAL] 零拷贝原则**: 对于局域网实际传输的文件块 (Byte Array)，**绝对禁止**通过 FFI 将字节流传递给 Dart。
* **执行边界**: 所有的 Socket 读写、文件切片、I/O 落盘，必须 100% 封闭在 KMP 内部完成。Dart 层仅通过 FFI 获取 `进度百分比 (Int)` 和 `速度 (Double)`。

## 🚀 给 AI 的系统级指令 (System Instructions for LLM)

1. **理解上下文**: 当用户要求你增加新功能时，首先判断该功能属于 **KMP Core** 还是 **Flutter UI**。
2. **遵守隔离**: 如果用户要求开发 UI 界面，你必须默认在 `MockCastApi` 中添加假数据接口，禁止直接修改 FFI 映射（除非用户显式要求“联调 FFI”）。
3. **内存安全**: 只要你在生成 Dart 代码时使用了 `Pointer<Utf8>` 或分配了原生内存，你生成的代码中**必须**包含对应的 `free()` 调用。
4. **拒绝 JVM**: 生成 KMP 代码时，禁止使用 `java.io.*`，`java.net.*`，请使用 `okio`, `ktor-network` 等 KMP 官方库。
5. **代码输出要求**: 不要解释基础概念。直接输出高内聚、符合上述设计模式的完整代码。
